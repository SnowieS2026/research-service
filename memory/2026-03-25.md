## Agent-to-Agent Pipeline Architecture (2026-03-25)

**COMPLETED:** Full rewrite of the bounty passive pipeline into a multi-agent system.

### New Files
- `src/agents/AgentMessage.ts` — typed message envelopes for all agent communication
- `src/agents/AgentQueue.ts` — persistent FIFO queue (JSONL files, no shared memory)
- `src/agents/BaseAgent.ts` — abstract base class with polling loop, retry logic, repair request support
- `src/agents/CoordinatorAgent.ts` — pipeline state machine (IDLE→DISCOVERY→BROWSER→SCANNING→REPORTER→IDLE)
- `src/agents/DiscoveryAgent.ts` — discovers programs across all platforms
- `src/agents/BrowserAgent.ts` — fetches program pages, computes diffs, self-heals Playwright crashes
- `src/agents/ScannerAgent.ts` — runs active scans (XSS, SQLi, SSRF, Auth, Nuclei, etc.)
- `src/agents/ReporterAgent.ts` — generates change reports and scan reports
- `src/agents/RepairAgent.ts` — autonomous repair specialist (browser crash, tool install, disk cleanup, rate limit handling)
- `src/agents/run.ts` — unified process entry point for all agents
- `src/agents/AgentRunner.ts` — (same as run.ts)

### Key Design
- **Message passing:** All agents communicate via `logs/agent-queue/<agentName>.queue.jsonl` (JSONL, append-only)
- **Async acks:** Messages acknowledged via `logs/agent-queue/<agent>.ack.<msgId>` sentinel files
- **No shared memory:** No concurrent access issues — each agent is single-threaded
- **Self-healing:** RepairAgent receives REPAIR:ISSUE from any agent, diagnoses (browser-crash/tool-missing/network/rate-limit/disk/parse), and sends REPAIR:DONE
- **Fast-track:** DiscoveryAgent sends new programs directly to BrowserAgent without waiting for full tick
- **Cron jobs updated:** Pipeline now runs via `npm run agent` (all agents in unified mode)

### Commands
- `npm run build && npm run agents` — build and run all agents
- `npm run build && node dist/src/index.js --agent` — same via index.ts
- `node dist/src/index.js --agent coordinator` — run single agent

### Old Files Still in Use
- `src/index.ts` — main entry, updated to support `--agent` mode alongside existing CLI modes
- `src/scanner/ParallelScanner.ts` — still used by ScannerAgent
- `src/storage/BountyDB.ts`, `SnapshotManager.ts`, `RunState.ts` — still used by agents

## Session 2026-03-25 11:18 — AdaptationAgent (ML Tool Strategy)

### What Was Built
New `AdaptationAgent` that learns which tools work best per bug bounty program and automatically adjusts the tool strategy for each scan run.

**Core logic:**
- Each scan run, the Coordinator sends `ADAPTATION:REVIEW` to AdaptationAgent for each discovered program
- AdaptationAgent replies with `ADAPTATION:STRATEGY` containing per-tool enable/disable decisions
- After each `SCAN:DONE`, Coordinator forwards results to AdaptationAgent to update tool scores
- Tool scoring: +2 per HIGH/CRITICAL finding, +1 per MEDIUM finding, -1 per false positive

**Confidence levels:**
- `low` (< 1 run): enable tools with score > 0 OR platform defaults
- `medium` (3+ runs): only enable tools with score > 0
- `high` (5+ runs): aggressive pruning — only score > 0 tools

**Always-on baseline:** `nuclei` is always enabled (low false positive rate, broad coverage).

**Files created/modified:**
- `src/agents/AdaptationAgent.ts` — new agent (ToolName scores, store, strategy builder)
- `src/agents/AgentMessage.ts` — added `adaptation` to AgentName, new message types

- `src/agents/CoordinatorAgent.ts` — queries AdaptationAgent pre-scan, forwards results post-scan
