## Agent-to-Agent Pipeline Architecture (2026-03-25)

**COMPLETED:** Full rewrite of the bounty passive pipeline into a multi-agent system.

### New Files
- `src/agents/AgentMessage.ts` ‚Äî typed message envelopes for all agent communication
- `src/agents/AgentQueue.ts` ‚Äî persistent FIFO queue (JSONL files, no shared memory)
- `src/agents/BaseAgent.ts` ‚Äî abstract base class with polling loop, retry logic, repair request support
- `src/agents/CoordinatorAgent.ts` ‚Äî pipeline state machine (IDLE‚ÜíDISCOVERY‚ÜíBROWSER‚ÜíSCANNING‚ÜíREPORTER‚ÜíIDLE)
- `src/agents/DiscoveryAgent.ts` ‚Äî discovers programs across all platforms
- `src/agents/BrowserAgent.ts` ‚Äî fetches program pages, computes diffs, self-heals Playwright crashes
- `src/agents/ScannerAgent.ts` ‚Äî runs active scans (XSS, SQLi, SSRF, Auth, Nuclei, etc.)
- `src/agents/ReporterAgent.ts` ‚Äî generates change reports and scan reports
- `src/agents/RepairAgent.ts` ‚Äî autonomous repair specialist (browser crash, tool install, disk cleanup, rate limit handling)
- `src/agents/run.ts` ‚Äî unified process entry point for all agents
- `src/agents/AgentRunner.ts` ‚Äî (same as run.ts)

### Key Design
- **Message passing:** All agents communicate via `logs/agent-queue/<agentName>.queue.jsonl` (JSONL, append-only)
- **Async acks:** Messages acknowledged via `logs/agent-queue/<agent>.ack.<msgId>` sentinel files
- **No shared memory:** No concurrent access issues ‚Äî each agent is single-threaded
- **Self-healing:** RepairAgent receives REPAIR:ISSUE from any agent, diagnoses (browser-crash/tool-missing/network/rate-limit/disk/parse), and sends REPAIR:DONE
- **Fast-track:** DiscoveryAgent sends new programs directly to BrowserAgent without waiting for full tick
- **Cron jobs updated:** Pipeline now runs via `npm run agent` (all agents in unified mode)

### Commands

## Session 2026-03-25 11:18 ‚Äî AdaptationAgent (ML Tool Strategy)

### What Was Built
New `AdaptationAgent` that learns which tools work best per bug bounty program and automatically adjusts the tool strategy for each scan run.

**Core logic:**
- Each scan run, the Coordinator sends `ADAPTATION:REVIEW` to AdaptationAgent for each discovered program
- AdaptationAgent replies with `ADAPTATION:STRATEGY` containing per-tool enable/disable decisions
- After each `SCAN:DONE`, Coordinator forwards results to AdaptationAgent to update tool scores
- Tool scoring: +2 per HIGH/CRITICAL finding, +1 per MEDIUM finding, -1 per false positive

**Confidence levels:**
- `low` (< 1 run): enable tools with score > 0 OR platform defaults
- `medium` (3+ runs): only enable tools with score > 0
- `high` (5+ runs): aggressive pruning ‚Äî only score > 0 tools

**Always-on baseline:** `nuclei` is always enabled (low false positive rate, broad coverage).

**Files created/modified:**
- `src/agents/AdaptationAgent.ts` ‚Äî new agent (ToolName scores, store, strategy builder)
- `src/agents/AgentMessage.ts` ‚Äî added `adaptation` to AgentName, new message types
- `src/agents/BaseAgent.ts` ‚Äî `setup()` changed from `protected` to `public`
- `src/agents/CoordinatorAgent.ts` ‚Äî queries AdaptationAgent pre-scan, forwards results post-scan
- `src/agents/ScannerAgent.ts` ‚Äî applies per-program tool configs from adaptation strategies
- `src/agents/run.ts` ‚Äî added AdaptationAgent to all-agent and single-agent runs
- Store: `logs/agent-state/adaptation-store.json` (persisted per-program tool scores)
